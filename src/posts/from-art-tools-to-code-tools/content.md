# From Art Tools to Code Tools

The more I learn programming, the more I realize I was already thinking like a tool builder.

Art tools taught me layers, masks, timing, texture, composition, iteration, and export settings. Code tools teach me state, functions, data flow, components, events, and build steps.

They are different languages, but they both care about structure.

Creative tech sits right in that overlap. A shader is visual, but it is also math. A game effect is aesthetic, but it is also timing and logic. A portfolio site is design, but it is also routing, content modeling, and deployment.

That bridge is exciting to me because it means programming does not have to replace art. It can support it.

I want to build little systems that help creative work exist more easily: portfolio templates, sticker galleries, blog structures, art generators, game prototypes, cozy interfaces, tiny tools with personalities.

Not everything has to be huge to be meaningful.

Sometimes the best tool is a small one that does one job clearly.

That is the version of software I am most interested in learning: soft enough to feel human, structured enough to hold.
